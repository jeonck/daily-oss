# My Context

> 이 문서는 수집된 오픈소스 프로젝트의 "행동 판정" 기준이다.
> 여기 없는 주제는 원칙적으로 "무관" 또는 "학습"으로 판정한다.

## 내 환경/현황

- IT 분야 전반에 관심이 있는 실무자/의사결정자 — 특정 스택에 고정되어 있지 않고,
  개발자·DevOps·AI·기획·프론트엔드·백엔드·시스템 관리자·인프라 전 영역을 폭넓게 살핀다.
- 새로 알려지고 있는(신규 공개, 스타 급증, 커뮤니티에서 화제) 오픈소스 프로젝트 중
  "실제로 써볼 가치가 있는 것"을 빠르게 발견하는 것이 목적.
- 직접 운영 중인 특정 프로덕션 스택은 없음 — 즉시조치는 "당장 사용/도입해볼 만큼 완성도와
  파급력이 높은 프로젝트"를 의미하며, 특정 인프라 장애 대응이 아니다.

## 관심 분야 (학습 판정 기준)

### 개발자 도구
- CLI, 에디터/IDE 확장, 빌드/테스트 도구, 언어/런타임, 개발 생산성 도구

### DevOps / 인프라 / 시스템 관리
- CI/CD, 컨테이너/오케스트레이션(Kubernetes 생태계), IaC, 모니터링/관측성,
  네트워킹, 클라우드 네이티브 인프라 도구

### AI / LLM
- LLM 프레임워크, AI 에이전트, RAG/벡터DB, 파인튜닝/추론 최적화, MCP 서버,
  AI 코딩 도구

### 프론트엔드 / 백엔드
- 웹 프레임워크, UI 컴포넌트/디자인 시스템, 상태관리, API 프레임워크,
  데이터베이스/ORM, 백엔드 아키텍처 도구

### 기획 / 협업
- 프로덕트/프로젝트 관리, 노코드-로우코드, 협업·문서화 도구,
  데이터 시각화/대시보드

### 명시적 제외 (관심 분야처럼 보여도 무관 처리)
- 단순 튜토리얼/블로그 홍보, 특정 회사 제품의 마케팅성 릴리즈 노트
- 게임, 투자/금융 정보 인덱스, 개인 일기·창작물 등 IT 실무 도구가 아닌 것
- 설명이 전혀 없고 README/용도를 전혀 짐작할 수 없는 저장소

## 소스 유형별 판정 원칙 (중요)

**핵심 목표는 "따라해 볼 만한 실제 오픈소스 저장소"를 찾아내는 것이다.**
GitHub 저장소(제목에 `owner/repo`와 ★스타 수가 포함된 항목)와, 뉴스/블로그 기사(도구를
"소개"하거나 "다룬" 글)를 다르게 취급한다.

- **GitHub 저장소 (github_search 소스, 제목이 `owner/repo (★N)` 형식)**
  - 설명이 있고 위 "관심 분야" 중 하나에 조금이라도 해당하면 **기본값은 학습이 아니라 백로그**다.
    "무관"은 설명이 전혀 없거나 명시적 제외 대상일 때만 사용한다.
  - 스타 500개 이상이거나, README/설명만으로 "지금 클론해서 5분 안에 뭘 하는지 확인할 수 있다"고
    판단되면 **즉시조치**로 올린다.
  - 액션은 항상 `git clone <url>` 또는 실행/설치 명령 등 구체적인 "써보는 행동"으로 쓴다.
- **뉴스/블로그 기사 (RSS 소스 대부분)**
  - 기사가 특정 오픈소스 프로젝트를 소개하고 있다면, 액션에 그 프로젝트의 **GitHub 저장소 URL**을
    직접 찾아 적고 "레포 클론해서 시도해보기" 형태로 쓴다. 프로젝트를 특정할 수 없는 일반 뉴스/의견
    기사(트렌드 해설, 인터뷰, 벤치마크 논쟁 등)는 학습 이상으로 올리지 않는다.
  - 같은 프로젝트를 다루는 기사와 GitHub 저장소 항목이 동시에 수집되면, 저장소 항목을 우선하고
    기사는 무관 또는 학습으로 낮춘다(중복 노출 방지).

## 판정 시 우선순위
1. 지금 클론해서 5분 안에 가치를 확인할 수 있는 완성도 높은 신규/급부상 오픈소스 저장소 → 즉시조치 후보
2. 관심 분야에 해당하는 유용한 오픈소스 저장소(당장 급하지 않거나 초기 단계) → 백로그 후보
3. 특정 저장소를 짚기 어려운 트렌드/기술 해설 기사 → 학습 후보
4. 그 외 일반 뉴스, 무관 주제, 마케팅성 글, 명시적 제외 항목 → 무관
